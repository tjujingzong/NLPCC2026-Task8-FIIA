# FIIA 迭代攻击流水线方案

## 一、目标

从 1089 条原始数据中，通过「生成 → 评估 → 筛选 → 精炼」循环迭代，最终产出 **200 条高质量攻击样本**（单条 Score ≥ 0.5），最大化 AttackScore。

---

## 二、核心指标回顾

| 指标 | 公式 | 说明 |
|------|------|------|
| **MIR** | \(1 - \frac{\max(c_T, c_F, c_U)}{10}\) | 10 轮推理不一致率，范围 [0, 0.6] |
| **Score_i** | \(\max(MIR_{attack} - MIR_{orig}, 0)\) | 单条攻击得分 |
| **AttackScore** | \(\sum Score_i\) | 总得分，理论最大 120（200 条 × 0.6） |

## 四、各阶段详细说明

### 4.1 Round 1 — 初始攻击生成与评估

| 步骤 | 说明 |
|------|------|
| **生成** | 对全部 1089 条数据，使用 `deepseek-v4-pro` 模型，调用攻击生成 Prompt（让步转折 + 多层模糊限定语），每条生成 1 个攻击变体 |
| **内联验证** | 每生成一条立即进行 R2-R4 核验；失败则将错误反馈给模型重新生成，最多重试 3 次；3 次后仍失败则放弃该条目 |
| **评估** | 对每条样本：① 原句 10 轮推理 → MIR_orig；② 攻击句 10 轮推理 → MIR_attack；③ 计算 Score |
| **筛选** | Score ≥ 0.5 → 合格池；Score < 0.5 → 低分池 |

**输出文件**：
- `01_initial_attacks.json` — 1089 条生成的攻击句（含策略标注）
- `02_eval_round1.json` — 全部 1089 条的评估结果 `{id, mir_orig, mir_attack, score, ...}`
- `03_qualified_round1.json` — 合格条目（Score ≥ 0.5）
- `04_lowscore_round1_sorted.json` — 低分条目（按攻击潜力排序）

---

### 4.2 低分池排序策略（"优先攻击容易攻击到的"）

排序规则（多级排序，从高到低）：

| 优先级 | 排序字段 | 方向 | 理由 |
|--------|----------|------|------|
| **P1** | `score`（当前得分） | 降序 | 得分越高说明攻击思路越接近成功，微调空间大 |
| **P2** | `mir_orig`（原句不一致率） | 降序 | MIR_orig 越高，说明模型对原句本身就不确定，更容易被扰动 |
| **P3** | `mir_attack - mir_orig`（提升幅度） | 降序 | 提升幅度越大说明当前策略方向有效 |

> 直观理解：优先攻击那些「已经有一定效果、且原句本身不稳定」的样本。

### 4.3 Round 2~4 — 迭代精炼

#### 精炼 Prompt 设计（关键）

与初始生成的 Prompt 不同，精炼阶段向 DeepSeek Pro 模型提供**完整上下文**：

```
【任务说明】你是一个红队攻击专家...

【原始数据】
- 原句：{text_original}
- 谓词：{predicate}
- 假设：{hypothesis}

【上次攻击尝试】
- 改编句：{previous_text_attack}
- 策略：{previous_strategy}

【评估反馈】
- 原句不一致率 MIR_orig：{mir_orig}（模型在原句上 {10-max_count} 次一致）
- 攻击句不一致率 MIR_attack：{mir_attack}（模型在攻击句上 {10-max_count} 次一致）
- 攻击得分 Score：{score}（目标 ≥ 0.5，差距 {0.5 - score}）

【请改进】
分析上次攻击的不足，给出一个全新的改编句。要求：
1. 保持 R2-R5 核验合规（谓词完整、小句保留≥70%、编辑距离≥65%、通顺度≥60%）
2. 改变策略角度（如上次用让步转折，这次可尝试信息稀释/情态转换/否定嵌入等）
3. 只输出策略和句子
```

#### 每轮精炼流程

1. **批量精炼**：对低分池全部条目，使用 DeepSeek Pro 模型重新生成攻击句
2. **重新评估**：对新攻击句做 10+10 轮推理，计算新 Score
3. **更新池**：新 Score ≥ 0.5 → 移入合格池；仍 < 0.5 → 留在低分池
4. **检查终止条件**

## 五、内联验证机制（生成即验证）

每次生成攻击句后，**立即进行 R2-R4 核验**（纯计算，无需加载模型）：

| 规则 | 检查内容 | 阈值 |
|------|----------|------|
| R2 | 谓词必须完整出现在攻击句中 | 子串包含 |
| R3 | 假设小句字符能按顺序在攻击句中找到 | LCS覆盖率 ≥ 0.7 |
| R4 | 与原句的编辑距离相似度 | ≥ 0.65 |

**重试机制**：
```
生成攻击句 → 验证 R2-R4
    └─ 通过 → 进入评估阶段
    └─ 失败 → 将错误反馈给模型，重新生成（最多 3 次）
              └─ 3 次后仍失败 → 放弃该条目，不进行攻击测试
```

这样可以在源头排除不合规样本，避免浪费评估 API 调用。

## 六、模型分工

| 阶段 | 模型 | 用途 | 原因 |
|------|------|------|------|
| 初始生成 | `deepseek-v4-pro` | 批量生成 1089 条攻击 | 更强的理解能力，提高一次通过率 |
| 评估推理 | `deepseek-v4-flash` | 10 轮一致性推理 | 与被攻击模型一致 |
| 迭代精炼 | `deepseek-v4-pro` | 基于反馈改进攻击 | 更强的理解和生成能力 |

## 八、终止条件与最终提交策略

### 8.1 迭代终止条件

| 条件 | 动作 |
|------|------|
| `len(qualified_pool) ≥ 200` | 立即停止迭代，进入最终提交 |
| 已完成 3 轮精炼（即 Round 4 评估后） | 停止迭代，进入最终提交 |
| 低分池为空 | 自然停止，进入最终提交 |

### 8.2 最终提交 200 条选取策略

为了**最大化 AttackScore**，最终提交的 200 条按以下优先级选取：

| 优先级 | 来源 | 排序方式 | 说明 |
|--------|------|----------|------|
| **P1** | 合格池（Score ≥ 0.5） | Score 降序 | 优先保留高分攻击（0.6 优于 0.5） |
| **P2** | 低分池（Score < 0.5） | Score 降序 | 合格池不足 200 时，按分数从高到低补齐 |

```
if 合格池 ≥ 200:
    → 取 Top 200 (按 Score 降序，0.6 > 0.5)
else:
    → 全部合格条目 + (200 - 合格数) 条低分池最高分条目
```

> 直观理解：总是用当前最好的 200 条攻击提交，不浪费任何一个提交名额。

**最终提交格式**（与 sample_submission.json 一致）：
```json
[
  {"id": "0048", "text_attack": "尽管…似乎…但…"},
  {"id": "0120", "text_attack": "或许…然而…"}
]
```

## 九、断点续传机制

由于全量 1089 条评估耗时长（约 6 小时/轮），脚本支持：

- **progress.json**：记录当前阶段、已完成的 item ID 列表
- 脚本启动时自动检测 `progress.json`，从中断点继续
- 每完成一个 batch（如 50 条）自动保存进度
- 支持 `--resume` 参数强制从断点恢复
- 支持 `--from-scratch` 参数忽略进度重新开始

## 十、运行方式

### 10.1 环境准备

```bash
# 1. 激活虚拟环境
source venv/bin/activate

# 2. 确认 .env 已配置 DeepSeek API Key
cat .env
# DEEPSEEK_API_KEY=sk-your-key-here
```

### 10.2 常用命令

```bash
# 完整运行（从头开始，1089 条全量）
python iterative_attack_pipeline.py

# 从断点恢复（网络中断后继续）
python iterative_attack_pipeline.py --resume

# 从断点恢复 + 自定义并行度
python iterative_attack_pipeline.py --resume --workers 5

# 完全重新开始（清空进度）
python iterative_attack_pipeline.py --from-scratch

# 自定义参数运行
python iterative_attack_pipeline.py \
  --target 200 \
  --threshold 0.5 \
  --max-refinement-rounds 3 \
  --batch-size 50 \
  --output-dir results-v2

# 查看帮助
python iterative_attack_pipeline.py --help
```

### 10.3 后台运行（推荐，避免终端断开导致中断）

```bash
# 使用 nohup 后台运行，日志输出到 pipeline.log
nohup python iterative_attack_pipeline.py > pipeline.log 2>&1 &

# 查看实时日志
tail -f pipeline.log

# 查看进度
tail -20 pipeline.log
```

### 10.4 预估耗时

| 阶段 | API 调用量 | 预估耗时 |
|------|-----------|----------|
| Round 1 生成 (含验证重试) | 1089×(1~4) 次 | ~30-60 分钟 |
| Round 1 评估 | 实际通过数×20 次 | ~4-6 小时 |
| 每轮精炼 (含验证重试) | ≤数量×(1+20+重试) 次 | ≤5-6 小时 |
| **总计（最坏情况）** | **~90,000 次** | **~20-24 小时** |

> 实际耗时取决于 API 响应速度和并发限制。断点续传机制确保即使中断也能从中断处继续。
